install → init → run → status

Get a local Stellar pipeline running in minutes.

flowctl is the orchestration and operations layer. Use nebu for rapid querying and processor prototyping. Use flowctl when you want a supervised pipeline with an embedded control plane, status visibility, and a repeatable operator workflow.

canonical first success
go install github.com/withobsrvr/flowctl@latest
export PATH="$HOME/go/bin:$PATH"
flowctl init --preset testnet-duckdb
flowctl run stellar-pipeline.yaml

The default DuckDB path downloads components automatically and does not require a local Docker daemon.

option 1

Go install

go install github.com/withobsrvr/flowctl@latest
export PATH="$HOME/go/bin:$PATH"
flowctl version

Best default if you already have Go installed.

option 2

Curl installer

curl -sSL https://flowctl.withobsrvr.com/install.sh | sh
flowctl version

Pulls a release binary from GitHub. No toolchain required.

option 3

Build from source

git clone https://github.com/withobsrvr/flowctl.git
cd flowctl
make build
./bin/flowctl version

Recommended for contributors working in this repository.

first pipeline

Start with DuckDB.

1. Generate and validate

flowctl init --preset testnet-duckdb
flowctl validate stellar-pipeline.yaml

This creates a `flowctl/v1` starter pipeline using process orchestration and auto-downloaded components.

2. Run and inspect

flowctl run stellar-pipeline.yaml

# in another terminal
flowctl status
flowctl pipelines active

The embedded control plane is the default. `status` and `pipelines active` are the core operator checks.

3. Query the local DuckDB file

duckdb stellar-pipeline.duckdb "SELECT event_type, COUNT(*) FROM contract_events GROUP BY event_type"

Stop the pipeline first if you want exclusive access to the file from the DuckDB CLI.

when to use what

flowctl vs nebu

  • flowctl: supervised pipelines, run tracking, health visibility, operational workflows.
  • nebu: rapid extraction, Unix pipes, local experimentation, processor prototyping.
  • Prototype with nebu, then productionize behind flowctl.
operator workflow

Canonical commands

flowctl init
flowctl validate stellar-pipeline.yaml
flowctl run stellar-pipeline.yaml
flowctl status
flowctl pipelines active
flowctl pipelines run-info <run-id>
flowctl pipelines stop <run-id>
troubleshooting

Common first-run issues

Port already in use

flowctl run --control-plane-port 0 stellar-pipeline.yaml

Or choose a fixed port and pass the same address/port to `flowctl status` and `flowctl pipelines active`.

Component download failed

flowctl run stellar-pipeline.yaml --log-level=debug

First run requires network access to fetch components into `~/.flowctl/`.

DuckDB CLI missing

The pipeline still works without the DuckDB CLI. Install `duckdb` separately if you want to query the generated file from your shell.

Status cannot connect

flowctl status --control-plane-address 127.0.0.1 --control-plane-port 9090

Use the same control plane address and port you started the pipeline with.
